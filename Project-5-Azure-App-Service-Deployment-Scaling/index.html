Walkthrough
Step 01
Create the App Service
  1. Navigate to App Services → + Create
  2. Name: globally unique (becomes yourname.azurewebsites.net)
  3. Runtime Stack: Node.js 20 LTS, Python 3.12, or .NET 8
  4. Pricing Tier: Standard S1 or higher — required for deployment slots and autoscaling
App Service basics — name, runtime stack, and pricing tier
App Service deployed and live at azurewebsites.net URL
Pricing tier matters: Free and Shared tiers don't support deployment slots or autoscaling. Standard S1+ is the minimum for production-like features.
Step 02
Deploy via GitHub Actions
  1. Push your application code to a GitHub repository
  2. App Service portal → Deployment Center → GitHub
  3. Authorize Azure to access your GitHub account
  4. Select repository and branch — Azure auto-generates a GitHub Actions workflow file
  5. Every push to the branch triggers an automatic deployment
Deployment Center connected to GitHub — CI/CD pipeline active
GitHub Actions workflow auto-generated and running on push
Step 03
Set Up Deployment Slots
  1. App Service → Deployment Slots → + Add Slot
  2. Name: staging, clone settings from Production
  3. Point a separate GitHub branch to the staging slot via Deployment Center
  4. Test at https://<app-name>-staging.azurewebsites.net
  5. When ready: Swap → promotes staging to production with zero downtime
Staging slot created — separate URL for pre-production testing
Slot swap — staging promoted to production atomically and reversibly
Step 04
Configure Autoscaling
  1. Navigate to App Service Plan → Scale Out (App Service Plan)
  2. Select Custom autoscale
  3. Scale Out rule: CPU > 70% for 10 min → increase count by 1
  4. Scale In rule: CPU < 30% for 10 min → decrease count by 1
  5. Instance limits: Minimum 1, Maximum 5, Default 1
Autoscale rules — scale out at 70% CPU, scale in at 30%
Instance limits set — min 1, max 5, default 1
Step 05
Configure Backup and Alerts
  1. App Service → Backups → Configure → link a Storage Account
  2. Scheduled: Daily, retention 7–30 days → Save
  3. Run Backup Now to confirm it completes
  4. App Service → Monitoring → Alerts → + New Alert Rule
  5. Signal: Average Response Time, condition: greater than 2 seconds
  6. Action Group: email notification → Save
Daily backup configured to Azure Storage — retention 30 days
Alert rule — triggers email when average response time exceeds 2 seconds
Azure Monitor metrics — CPU, response time, requests, and HTTP 5xx errors
Key Concepts
PaaS vs. IaaS
App Service (PaaS) abstracts the OS and runtime — you manage only the application code and config. VMs (IaaS) give you full control but require you to patch, secure, and maintain the OS.
Deployment Slots
Separate instances sharing the same App Service Plan. A slot swap is atomic and reversible — if the new version breaks, swap back. Standard blue/green deployment practice.
Autoscaling vs. Scale Up
Autoscaling (scale out) adds more instances horizontally. Scale up increases the size of the existing instance. Autoscaling is more resilient and cost-efficient for variable load.
GitHub Actions for Azure
Azure publishes official GitHub Actions for App Service, Functions, Container Apps, and more. Auto-generated from Deployment Center — the most common way modern Azure deployments are automated.
