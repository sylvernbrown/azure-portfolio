Walkthrough
Step 01
Create the Virtual Network

Created an Azure Virtual Network to provide network isolation and communication between Azure resources. Configured two subnets to logically separate application and database resources while allowing secure communication within the VNet.

  • Navigate to Azure Portal → Virtual Networks → + Create
  • Resource group: rg-proj2-network-storage, VNet name: vnet-proj2, Region: West US
  • Address space: 10.0.0.0/16
  • Add Database subnet: 10.0.0.0/27, enable Private Subnet, add Service Endpoint: Microsoft.Storage
  • Add Web subnet: 10.0.1.0/27
  • Review + Create

Both subnets configured — Database (10.0.0.0/27) and Web (10.0.1.0/27) within the 10.0.0.0/16 address space
Why /27? A /27 gives 32 addresses (30 usable after Azure reserves 5). Right-sized for isolated workloads — not wasteful, not restrictive. Subnets enforce a security boundary: a web server should never have direct database-level network access.
Step 02
Configure a Network Security Group (NSG)

Created and associated a Network Security Group with the Web subnet. Configured inbound security rules to allow only HTTPS traffic while restricting all other inbound internet access.

  • Navigate to Network Security Groups → + Create
  • Name: nsg-web, Resource group: rg-proj2-network-storage
  • After deployment: NSG → Settings → Subnets → + Associate → select vnet-proj2 / Web
  • Add inbound rule: Port 443, Protocol TCP, Action Allow, Priority 100, Name Allow-HTTPS
  • Add inbound rule: Port *, Protocol Any, Action Deny, Priority 200, Name Deny-All-Inbound

nsg-web deployed — navigated to Settings → Subnets to associate with the Web subnet
Associate subnet — selected vnet-proj2 and Web subnet to attach the NSG
Inbound security rules showing default Azure rules: AllowVnetInBound (65000), AllowAzureLoadBalancerInBound (65001), DenyAllInBound (65500)
Adding Allow-HTTPS inbound rule — destination port 443, protocol TCP, action Allow, priority 100
Adding Deny-All-Inbound rule — all ports, any protocol, action Deny, priority 200 — blocks all other inbound traffic
Default rules cannot be deleted — but they can be overridden with a lower priority number. The default DenyAllInBound sits at priority 65500. Custom rules at 100 and 200 evaluate first, ensuring only HTTPS passes through while everything else is dropped.
Step 03
Deploy a Secure Storage Account

Created an Azure Storage Account with public network access disabled and a Private Endpoint connected to the Database subnet — restricting storage access to resources within the virtual network only.

  • Navigate to Storage Accounts → + Create
  • Performance: Standard, Redundancy: Locally Redundant Storage (LRS)
  • Networking tab: set Public network access to Disable
  • Add Private Endpoint: name storage-private-endpoint, sub-resource blob, VNet vnet-proj2, Subnet Database
  • Enable Private DNS integration: privatelink.blob.core.windows.net
  • Encryption: Microsoft-managed keys (default)

Storage account deployed — navigated to Security + networking → Encryption to verify encryption settings
Encryption type: Microsoft-managed keys — data encrypted at rest by default on all Azure Storage accounts
Private Endpoint vs. Service Endpoint: Service Endpoints route traffic over Microsoft's backbone but the storage account still has a public IP. A Private Endpoint assigns a private IP from inside your VNet — the storage account is fully air-gapped from the public internet. Much stronger isolation for sensitive data.
Step 04
Export and Deploy ARM Templates

Exported ARM templates from deployed Azure resources and imported them into Template Specs — demonstrating Infrastructure as Code concepts and enabling repeatable, version-controlled deployments.

  • Go to vnet-proj2 → Automation → Export template
  • Download the ARM template JSON
  • Navigate to Template Specs → Import template
  • Upload storage-template.json and configure as a new Template Spec
  • Name: Storage-Template, Version: 1
  • Confirm Template Spec appears in the gallery ready to deploy

Why Template Specs? Exporting templates from deployed resources captures the exact configuration and makes it version-controlled and repeatable. This is foundational IaC — instead of clicking through the portal again, you deploy the spec and get an identical environment every time.
Step 05
Configure Shared Access Signatures (SAS)

Generated Shared Access Signatures for the storage account to enable secure delegated access. Validated the configuration by connecting to the storage account via Azure Storage Explorer using the SAS URL.

  • Click Generate SAS and connection string
  • Open Azure Storage Explorer → Connect → Shared access signature URL (SAS)

SAS Token best practices: Always generate with the minimum required permissions and the shortest viable expiry. Never use account keys in production — they grant full access. SAS tokens are scoped and time-limited, making them far safer for delegated access scenarios.
Key Concepts
Network Segmentation
Separating web and database traffic into distinct subnets limits blast radius. Web servers and database resources operate in isolated network segments — a compromised web tier cannot reach storage directly.
Private vs. Service Endpoints
Service Endpoints route traffic over Microsoft's backbone but storage keeps a public IP. Private Endpoints assign a private IP from inside the VNet — the storage account is fully air-gapped from the public internet.
ARM Templates & Template Specs
JSON-based IaC for Azure. Exported directly from deployed resources and stored as Template Specs — enabling version-controlled, repeatable deployments. The foundation for automation pipelines and consistent environments.
Shared Access Signatures
Time-limited, scoped credentials for storage access. Grant minimum required permissions with the shortest viable expiry. Never expose account keys — SAS tokens are the secure alternative for delegated access.
