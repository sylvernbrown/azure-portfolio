Walkthrough
Step 01
Create Users and Groups
  1. Navigate to Microsoft Entra ID → Users → + New user
  2. Create at least two users — one for Admin group, one for standard User group
  3. Navigate to Groups → + New group
  4. Create App-Admins and App-Users security groups
  5. Add users to appropriate groups
New user created in Entra ID
App-Admins and App-Users security groups configured
Step 02
Register an Application
  1. Navigate to Entra ID → App registrations → + New registration
  2. Name: lab-app-01, Supported account types: Single tenant
  3. After creation, note the Application (client) ID and Directory (tenant) ID
  4. Go to Certificates & secrets → + New client secret — copy the value immediately
  5. Go to API permissions → + Add a permission → Microsoft Graph
  6. Add: User.Read and Group.Read.All → Grant admin consent
App registration created — Client ID and Tenant ID noted
Client secret generated — value copied immediately (only shown once)
Microsoft Graph permissions granted — User.Read and Group.Read.All with admin consent
Step 03
Assign Users via Enterprise Applications
  1. Navigate to Entra ID → Enterprise applications → lab-app-01
  2. Go to Users and groups → + Add user/group
  3. Assign App-Admins and App-Users groups to the application
Groups assigned to the enterprise application via Enterprise Applications blade
Step 04
Configure Conditional Access — Require MFA
  1. Navigate to Entra ID → Security → Conditional Access → + New policy
  2. Name: Require MFA — All Users
  3. Assignments: Users → All users
  4. Target resources: All cloud apps
  5. Access controls → Grant → Require multifactor authentication
  6. Enable policy → Save
Conditional Access policy — MFA required for all users on all cloud apps
Grant control set to Require multifactor authentication
Test in Report-only mode first — enabling a CA policy immediately on all users can lock people out. Report-only mode shows what would have been blocked without enforcing it.
Step 05
Review Sign-in Logs
  1. Navigate to Entra ID → Monitoring → Sign-in logs
  2. Filter by user, application, or status
  3. Review authentication method used, location, device, and conditional access result
Sign-in logs — authentication activity showing MFA enforcement results
Sign-in detail — Conditional Access policies applied and outcome
Key Concepts
App Registration vs. Enterprise App
App Registration is where you define the application identity — client ID, secrets, permissions. Enterprise Application is the service principal — where you manage who can access it.
Conditional Access
If-then policies: if a user matches conditions (location, device, app), then enforce controls (MFA, block, compliant device). The primary tool for zero-trust enforcement in Entra ID.
Microsoft Graph Permissions
Delegated permissions act on behalf of a signed-in user. Application permissions act as the app itself with no user context. Least-privilege: only request what the app actually needs.
Sign-in Logs
Every authentication attempt is logged — user, app, location, device, CA result, MFA method. Retained 30 days by default. Export to Log Analytics for longer retention and KQL queries.
